诊断并修复 bug。输入：$ARGUMENTS

输入可以是：bug 描述文字、日志文件路径、错误截图路径、或以上任意组合。

**核心原则**：
1. **诊断先行**：不理解根因之前禁止动代码。
2. **证据驱动**：每个结论必须有日志/代码/数据支撑，禁止猜测。
3. **全局视野**：修一个 bug，扫一类 bug。

---

## 阶段 1：证据收集

### 1a. 解析输入

- 如果输入是**文件路径** → 读取文件内容（日志、截图、crash report）
- 如果输入是**文字描述** → 提取关键信息：错误信息、触发条件、影响范围
- 如果输入是**截图** → 读取图片，识别错误信息和状态

### 1b. 日志取证（如有日志）

逐行分析日志，标注：
```
[时间线]     每条日志的相对时间顺序
[正常]       符合预期的行为
[异常]       不符合预期的行为 ← 重点关注
[关键转折]   从正常到异常的转折点 ← 根因线索
[重复模式]   多次出现的相同错误 ← 可能是循环/重试
```

输出格式：
```
=== 日志时间线分析 ===

第 1-5 行: [正常] 初始化成功
第 6 行:   [关键转折] 检测到异常状态
第 7-16 行: [异常] 错误详情
  → 问题：为什么会发生？

根因假设：
  假设 A: <描述> — 置信度 70%
  假设 B: <描述> — 置信度 20%
  假设 C: <描述> — 置信度 10%
```

### 1c. 检查已知问题

- 搜索 git log 最近 20 条 commit，看是否有相关的近期改动引入
- 搜索 `CLAUDE.md` 和 `README.md` 中的已知约束
- 搜索 issues（如有）

## 阶段 2：根因追踪

### 2a. 代码路径追踪

从错误信息出发，**反向追踪**到源代码：

1. **Grep 错误信息** → 找到输出该信息的代码位置
2. **向上追溯调用链** → 谁调用了这个函数？什么条件下走到这个分支？
3. **画调用链图**：
```
module_a.py:42 main_function()
  → module_b.py:64 process()
    → module_c.py:73 execute()
      → 异常抛出点 ← 错误注入点
```
4. **找到错误注入点** → 这就是根因所在

### 2b. 验证假设

对阶段 1 的每个假设，在代码中找证据：

```
假设 A: <描述>
  证据：读 xxx 代码 → 发现 ...
  结论：✅ 确认 / ❌ 排除 / ⚠️ 部分确认
```

**必须排除所有假设直到只剩一个，或找到确切的根因。**

### 2c. 输出根因诊断报告

```
=== 根因诊断 ===

错误现象：<什么出了问题>
触发条件：<什么场景下发生>
错误注入点：<文件:行号>
根因：<为什么会发生，一句话>

调用链：
  <完整的调用链图>

影响范围：
  - 直接影响：<什么功能不可用>
  - 间接影响：<其他可能受影响的模块>
```

## 阶段 3：复现

- 如果能写测试复现 → 写一个红灯测试作为验收标准
- 如果是环境/时序相关无法测试复现 → 在代码中标注触发条件，写注释说明为什么无法测试复现

## 阶段 4：全局扫描

**在修之前，先扫描同类问题**：

1. Grep 错误注入点的代码模式 → 其他文件有同样的模式吗？
2. 这个 bug 的根因是一个**通用模式**（如"没有超时保护"、"没检查 None"、"没有 retry"）→ 全局搜索同类
3. 列出所有发现的同类问题，一起修

## 阶段 5：修复

- 修当前 bug + 阶段 4 发现的所有同类问题
- **Two Hats 原则**：
  - 需要重构才能修 → 先小步重构（每步跑测试→绿→wip commit）
  - 再修 bug（跑测试→绿→wip commit）
  - 两件事不混着做
- [⚠️ 失败恢复] 重构时测试变红 → 立即 `git revert` 到上一个绿灯，不要 debug

## 阶段 6：补充测试

- 复现测试变绿 ✓
- 补充边界 case 测试
- 重点覆盖错误注入点附近的路径
- 三类必须覆盖：正常路径 + 边界 case + 错误路径

## 阶段 7：Code Review

执行 `/selfreview` 的 4 维度检查（正确性→健壮性→架构→接口质量），特别关注：
- 修复是否引入了新问题？
- 影响范围比预期的大吗？
- 对照阶段 2c 的"间接影响"列表，逐个确认没有破坏

## 阶段 8：验证

- 跑全量测试（`python -m unittest discover tests` 或 `pytest`），必须全部通过
- [⚠️ 失败恢复] 测试失败 → 回到阶段 5 重新修复

## 阶段 9：提交

🛑 **主动向用户汇报并询问是否提交。**

- commit（Conventional Commits，`fix:` 前缀）

---

## 失败处理

- **日志不够判断根因** → 在代码关键路径加诊断日志（logging + 上下文变量），让用户复现一次拿更详细的日志，不要猜
- **无法复现** → 加防御性代码 + 日志，标记为"未确认复现"，不要盲改
- **修复引入新失败** → 回到阶段 5，不要跳过 review
- **重构测试变红** → 立即 revert，尝试更小的步骤
- **根因在第三方库** → 记录 workaround，开 issue 到上游
